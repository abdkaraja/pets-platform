# Phase 2 — UI Design Contract

> Visual and interaction contract for the E-Commerce Module. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Tailwind CSS v4 utility classes — no component library) |
| Preset | not applicable |
| Component library | none — hand-built Razor partials + Tailwind utilities |
| Icon library | inline SVG only (no icon library dependency) |
| Font | System font stack (`font-sans`: system-ui, -apple-system, Segoe UI, sans-serif) |
| CSS framework | Tailwind CSS v4 via `@import "tailwindcss"` in `wwwroot/css/site.css` |
| JS framework | jQuery (slim, v3.x) + vanilla JS for Stripe.js integration |
| Layout engine | RTL (`<html lang="ar" dir="rtl">`) — all spacing uses logical properties |

### RTL Layout Contract (Critical)

The site renders in Arabic RTL direction. **All spacing must use Tailwind logical properties:**

| Directional Concept | Use Instead | Example |
|---------------------|-------------|---------|
| Left margin/padding | `ms-*` (margin-start) | `ms-2` not `ml-2` |
| Right margin/padding | `me-*` (margin-end) | `me-4` not `mr-4` |
| Text align left | `text-start` | `text-start` not `text-left` |
| Text align right | `text-end` | `text-end` not `text-right` |
| Float left | `float-start` | — |
| Border left | `border-s` | `border-s` not `border-l` |

**Exception:** `ml-auto` / `mr-auto` → use `ms-auto` / `me-auto`. Flexbox `justify-start`/`justify-end` are direction-safe.

## Spacing Scale

Standard Tailwind v4 default scale (matches existing Phase 1 usage):

| Token | Tailwind Class | Value | Usage |
|-------|---------------|-------|-------|
| xs | `p-1` / `m-1` / `gap-1` | 4px | Icon gaps, inline badge padding, tight spacing |
| sm | `p-2` / `m-2` / `gap-2` | 8px | Compact element spacing, input padding, card inner gaps |
| md | `p-4` / `m-4` / `gap-4` | 16px | Default element spacing, card padding, section gaps |
| lg | `p-6` / `m-6` / `gap-6` | 24px | Section padding, card-to-card gaps in grids |
| xl | `p-8` / `m-8` / `gap-8` | 32px | Major section breaks, page content padding |
| 2xl | `p-12` / `m-12` | 48px | Hero sections, major vertical breaks |
| 3xl | `p-16` / `m-16` | 64px | Page-level vertical spacing |

**Exceptions:**
- Cart item row vertical padding: `py-3` (12px) — compact list layout
- Product card image height: `h-48` (192px) — consistent image grid
- Product detail image: `h-64` to `h-96` (256–384px) — larger detail view
- Admin table row height: `py-2` (8px) — data-dense table rows
- Stripe Payment Element mount point: `min-h-[200px]` — Stripe requires minimum height

## Typography

| Role | Tailwind Class | Size | Weight | Line Height | Usage |
|------|---------------|------|--------|-------------|-------|
| Heading | `text-2xl font-bold` | 24px | 700 | 1.3 | Page titles, section headings, card titles, product names, cart total, empty state headings |
| Body | `text-base` | 16px | 400 | 1.5 | Default text, descriptions, form labels, metadata, product card prices |
| Helper | `text-sm` | 14px | 400 | 1.5 | Sidebar nav, search inputs, pagination, variant buttons, cart item details, error displays, admin table headers, admin buttons, form validation, inline stock editing, field-level errors, date text |
| Caption | `text-xs` | 12px | 400 | 1.4 | Badges, helper text, legal text, stock indicators, timestamps |

**Font weights used:** 400 (regular), 700 (bold) — exactly 2.

**Size migration notes (from checker audit):**
- `text-lg` (18px) used for product card prices → **migrated to `text-base` (16px)** — prices use `text-base font-bold` instead
- `text-xl` (20px) used for cart total and empty state headings → **migrated to `text-2xl` (24px)** — aligns with Heading role
- `text-sm` (14px) used in 17+ locations across inputs, sidebar nav, pagination, variant buttons, cart details, error displays, admin table headers, form validation, and inline editing — distinct "input/helper" role that `text-xs` (12px) is too small for

**RTL text rules:**
- Headings: `text-start` alignment (RTL-aware)
- Price display: Always LTR (`dir="ltr"`) — currency formatting is LTR even in RTL pages
- Product names: Inherit RTL direction (Arabic text)
- Admin tables: `text-start` for all data cells

### Page 1: Product Catalog (`Catalog/Index`)

**Route:** `/Catalog` or `/Catalog?search=...&category=...&petType=...&brand=...&minPrice=...&maxPrice=...&sort=...&page=1`

**Layout:** `_Layout.cshtml`

**Focal Point:** The product image grid is the primary visual anchor — it occupies ~70% of the viewport width and contains the largest visual elements (product images). The filter sidebar is secondary (240px, neutral background, text-based controls). Visual hierarchy: product images first → prices/CTA second → filters third.

**Structure:**

```
┌──────────────────────────────────────────────────────────────┐
│ Page Title: "Shop" (text-2xl font-bold)                      │
├──────────────┬───────────────────────────────────────────────┤
│              │                                               │
│   Filters    │           Product Grid                        │
│   Sidebar    │                                               │
│   (240px)    │   ┌─────────┐ ┌─────────┐ ┌─────────┐       │
│              │   │ Product │ │ Product │ │ Product │       │
│  Search      │   │  Card   │ │  Card   │ │  Card   │       │
│  ─────       │   └─────────┘ └─────────┘ └─────────┘       │
│  Pet Type    │   ┌─────────┐ ┌─────────┐ ┌─────────┐       │
│  ─────       │   │ Product │ │ Product │ │ Product │       │
│  Category    │   │  Card   │ │  Card   │ │  Card   │       │
│  ─────       │   └─────────┘ └─────────┘ └─────────┘       │
│  Price Range │                                               │
│  ─────       │           Pagination                          │
│  Brand       │   [1] [2] [3] ... [N]                        │
│              │                                               │
└──────────────┴───────────────────────────────────────────────┘
```

**Components:**

**Filter Sidebar (left, 240px, `bg-white rounded-lg shadow p-4`):**
- Search input: `<input type="search" placeholder="Search products..." class="w-full border border-gray-300 rounded-lg px-3 py-2 text-sm">`
- Pet Type filter: Radio buttons or clickable chips — "All", "Dogs", "Cats", "Birds", "Fish", "Small Animals"
- Category filter: Checkboxes for categories (loaded from DB)
- Price Range: Two inputs (min/max) — `<input type="number" placeholder="Min" class="w-full border border-gray-300 rounded px-3 py-2 text-sm">`
- Brand filter: Checkboxes for brands (loaded from DB)
- "Clear Filters" link at bottom
- Filter state preserved in URL query params (server-side filtering, page reload)

**Product Card (`bg-white rounded-lg shadow hover:shadow-lg transition-shadow overflow-hidden`):**
- Image: `<img class="w-full h-48 object-cover">` or gray placeholder if no image
- Content padding: `p-4`
- Product name: `text-base font-bold mb-1` (truncated to 2 lines: `line-clamp-2`)
- Category/Brand: `text-xs text-gray-500 mb-2`
- Price: `text-base font-bold text-gray-900` — show computed price (base × multiplier for cheapest variant)
- Price range if variants differ: `text-base font-bold text-gray-900` — show "{min} – {max}"
- Stock badge (if ≤ 5): `text-xs text-yellow-600` — "Only {N} left"
- Out of stock badge: `text-xs text-red-600 font-bold` — "Out of Stock"
- Entire card is a link to `/Catalog/Details/{id}`

**Responsive:**
- Desktop (≥1024px): Sidebar + 3-column grid
- Tablet (768–1023px): Sidebar collapses to horizontal filter bar + 2-column grid
- Mobile (<768px): Filters collapse to "Filters" button that opens a slide-out panel + single column

**Sorting:**
- Dropdown above grid: `Sort by: Featured ▼` — options: Featured (default), Price: Low to High, Price: High to Low, Name: A–Z, Newest

**Pagination:**
- Below grid: `flex justify-center gap-2`
- Page buttons: `w-8 h-8 rounded border text-sm` — active: `bg-blue-600 text-white border-blue-600`
- Show: Previous / {1} {2} {3} ... / Next

## Interaction Patterns

### Add to Cart Flow

1. User clicks "Add to Cart" on product detail or product card
2. **If variant product:** Product card click → navigates to detail page (user must select variant first)
3. **If single-variant / no-variant product:** Product card has inline "Add to Cart" button
4. Button shows "Adding..." with spinner during AJAX POST
5. **Success:** Toast notification "Added to cart!" (auto-dismiss 3s), cart badge count updates
6. **Failure (out of stock):** Error message inline: "This item is out of stock."
7. **Failure (exceeds stock):** Error message: "Only {N} items available. You have {M} in your cart."

### Checkout Flow

1. User clicks "Proceed to Checkout" → navigates to `/Checkout`
2. User fills shipping info (name, address, city, phone)
3. Server creates Stripe Checkout Session with `ui_mode: "elements"`
4. Stripe Payment Element renders in the page (card form)
5. User enters card details, clicks "Pay ${amount}"
6. Stripe.js processes payment client-side (card data never touches server)
7. **On success:** Redirect to `/Checkout/Confirmation?session_id=...`
8. **On failure:** Error message below payment form, user can retry
9. Stripe webhook (`checkout.session.completed`) confirms payment server-side
10. If webhook is slow: Confirmation page shows "Your order is being confirmed..." with auto-refresh
11. After confirmation: Cart is cleared (per D-14)

### Search / Filter Flow

1. Filter changes trigger page reload with query params (server-side filtering)
2. Search input: debounced 300ms (jQuery) before submit
3. Filter state preserved in URL — shareable/bookmarkable
4. "Clear Filters" resets all filters to defaults
5. Results update with page reload (no AJAX for simplicity in v1)
6. Empty results: show "No products found" with "Clear Filters" link

### Quantity Update in Cart

1. User clicks +/- buttons
2. jQuery updates displayed quantity immediately (optimistic)
3. AJAX POST to update quantity on server
4. **If stock exceeded:** Server returns error, client reverts quantity, shows error
5. Cart total updates in order summary section

### Admin Status Update

1. Admin selects next status from dropdown
2. Clicks "Update Status"
3. Simple `confirm()` dialog: "Mark order as {status}?"
4. POST to server
5. Page reloads with success toast
