# Plan 04-01: Domain Entities + Database Schema + Service Layer Foundation

## Objective

Build the complete lost pets module foundation: domain entities with status lifecycle, database schema via EF Core configurations, all DTOs, the ILostPetService interface and implementation with matching logic, FluentValidation validators, and DI registration. After this plan, the lost pets module is fully functional at the service layer with a migration ready.

**Purpose:** Establishes the data model and business logic that the UI layers (Plans 02 and 03) consume. All domain rules (report lifecycle, match detection, photo management, authorization) are enforced here.

**Output:** Working service layer with synchronous matching, migration applied, DI wired up.

- **type**: create
  **files**: [src/PetPlatform.Application/Services/LostPetService.cs]
  **action**: Implement `LostPetService` following `PetService.cs` and `AdoptionService.cs` patterns:

    **Constructor**: Inject `IApplicationDbContext`, `IFileStorageService`, `IValidator<CreateLostPetReportDto>`, `IValidator<UpdateLostPetReportDto>`. Use `Guard.Against.Null` for each.

    **SearchReportsAsync**: Query `LostPetReports` where Status == Open (per D-16 all users see open reports). Apply optional filters:
    - ReportType: exact match (if provided)
    - Species: exact match
    - Breed: Contains, case-insensitive (if provided)
    - Color: Contains, case-insensitive (if provided)
    - Location: Contains, case-insensitive per D-11
    - DateFrom/DateTo: range on DateReported
    - SearchTerm: Contains on Description
    Include Photos. Apply pagination. Return `PagedResultDto<LostPetReportDto>`.

    **GetReportByIdAsync**: Find report by Id with Photos and optional Pet included. Return null if not found. No auth required (public per D-16).

    **CreateReportAsync**: Validate DTO. If `dto.PetId.HasValue`, verify Pet exists and belongs to reporter (query `_context.Pets` where Id == dto.PetId and OwnerId == reporterUserId — per Pitfall 3 from research). Create `LostPetReport.Create(...)`. Add to context. SaveChangesAsync to get the Id. Process photos: if photos provided and count between 1-5, call `_storageService.SaveFileAsync` for each and create `LostPetReportPhoto.Create(report.Id, photoPath)`, add to context. If photos count is outside 1-5 range, return `Result.Failure("You must upload between 1 and 5 photos.")` per Pitfall 4. SaveChangesAsync again. Then call `FindAndCreateMatchesAsync(report)` synchronously per D-08. SaveChangesAsync for notifications. Return mapped DTO.

    **UpdateReportAsync**: Validate DTO. Find report. Verify `report.ReporterUserId != reporterUserId` returns Failure per D-13/Pitfall 5. Verify `report.Status == Open` returns Failure (only Open can be edited per D-15). Call `report.UpdateDetails(...)`. If photos provided, remove existing photos (set collection), add new ones. Save. Return mapped DTO.

    **ResolveReportAsync**: Find report. Verify `report.ReporterUserId != reporterUserId` returns Failure per D-13. Call `report.Resolve()` (domain validates Open state). Save. Return success.

    **GetMyReportsAsync**: Query where `ReporterUserId == reporterUserId`. Include Photos. Order by CreatedAt desc. Return mapped list.

    **GetMyNotificationsAsync**: Query `MatchNotifications` where `ReporterUserId == userId`. Include `MatchedReport`. Order by CreatedAt desc. Return mapped list.

    **MarkNotificationReadAsync**: Find notification. Verify `notification.ReporterUserId != userId` returns Failure. Call `notification.MarkAsRead()`. Save. Return success.

    **Private FindAndCreateMatchesAsync(LostPetReport newReport)**: Per D-06 (same species + contains location) and D-08 (synchronous on creation):
    - Determine oppositeType (Lost→Found, Found→Lost)
    - Query for opposite-type open reports where `Species == newReport.Species` AND `Location.Contains(newReport.Location)` AND `ReporterUserId != newReport.ReporterUserId`
    - For each match, check no existing notification with same TriggeredReportId + MatchedReportId pair (Pitfall 1: deduplication)
    - Create `MatchNotification.Create(match.Id, newReport.Id, match.ReporterUserId, message)` for notifying existing reporters about new report
    - Also create `MatchNotification.Create(newReport.Id, match.Id, newReport.ReporterUserId, message)` for notifying new reporter about existing match (bidirectional)
    - Deduplicate both directions

    **Private MapToDto(LostPetReport report)**: Static mapper following AdoptionService pattern. Maps report properties, flattens Pet name/photo if linked, maps Photos collection to `List<LostPetReportPhotoDto>`.

    **Private MapToNotificationDto(MatchNotification notification)**: Static mapper. Flattens MatchedReport species/location/description for display context.
  **verify**: Service compiles, all business rules implemented, ownership checks present in every mutation method
  **acceptance_criteria**:

    - CreateReport enforces 1-5 photo count in service layer (Pitfall 4)
    - CreateReport triggers synchronous matching (D-08) with deduplication (Pitfall 1)
    - Matching queries opposite-type open reports with same species + location contains (D-06)
