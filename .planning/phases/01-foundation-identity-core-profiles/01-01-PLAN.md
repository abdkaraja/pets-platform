2. `AddIdentity<ApplicationUser, ApplicationRole>()` with options: `options.SignIn.RequireConfirmedAccount = true` (Pitfall #6 — email verification enforced), `options.Lockout.MaxFailedAccessAttempts = 5`, `options.Lockout.DefaultLockoutTimeSpan = TimeSpan.FromMinutes(15)` (AUTH-05), `options.Password.RequireDigit = true`, `options.Password.RequiredLength = 6`, `options.Password.RequireNonAlphanumeric = true`, `options.Password.RequireUppercase = true`. Chain `.AddEntityFrameworkStores<ApplicationDbContext>().AddDefaultTokenProviders().AddDefaultUI()`
3. `AddControllersWithViews()`
4. `AddAuthorizationBuilder()` — register policies: `"Permission:Users.View"` (RequireClaim "Permission" "Users.View"), `"Permission:Users.Manage"` (RequireClaim "Permission" "Users.Manage"), `"Permission:Roles.Create"` (RequireClaim "Permission" "Roles.Create"), `"Permission:Roles.Assign"` (RequireClaim "Permission" "Roles.Assign"), `"Permission:Pets.Manage"` (RequireClaim "Permission" "Pets.Manage"). Per RESEARCH.md Pattern 2.
5. `AddScoped<EmailSender>()` — register the IEmailSender<ApplicationUser> implementation

1. `UseExceptionHandler` ("/Home/Error")
2. `UseHttpsRedirection()`
3. `UseStaticFiles()`
4. `UseRouting()`
5. `UseAuthentication()` — BEFORE Authorization
6. `UseAuthorization()`
7. Area route: `app.MapControllerRoute(name: "areas", pattern: "{area:exists}/{controller=Home}/{action=Index}/{id?}")`
8. Default route: `app.MapControllerRoute(name: "default", pattern: "{controller=Home}/{action=Index}/{id?}")`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Host.MVC | Untrusted user input crosses into server-side MVC controllers |
| Host.MVC → Application | Service layer call — trust boundary for business logic enforcement |
| Application → Infrastructure | Data access boundary — EF Core maps entities to database |
| Identity middleware pipeline | Authentication/Authorization checks before controller execution |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Elevation of Privilege | Admin area controllers | critical | mitigate | Policy-based authorization on every Admin controller action per RESEARCH.md Pitfall #7 |
| T-01-02 | Tampering | Identity registration/login forms | high | mitigate | Anti-forgery tokens on all POST forms, server-side validation |
| T-01-03 | Information Disclosure | SeedData admin credentials | medium | accept | Default admin password Admin@12345! is for dev only; document in README that production must change |
| T-01-04 | Elevation of Privilege | Claims-based permissions | high | mitigate | SeedData assigns permissions only to appropriate roles; authorization policies enforce at controller level |
| T-01-05 | Denial of Service | Failed login brute force | medium | mitigate | Account lockout after 5 failed attempts with 15-minute lockout window (AUTH-05) |
| T-01-06 | Tampering | npm package installation | high | mitigate | All packages from Package Legitimacy Audit are [Approved] — no [ASSUMED] or [SUS] packages |
</threat_model>
