Build the Admin Area with three controllers. Every controller action uses policy-based authorization per RESEARCH.md Pattern 2 and Pitfall #7 (Admin Area Accessible Without Authorization).

**Create Area directory structure:**
Create `src/PetPlatform.Host.MVC/Areas/Admin/Controllers/` and `Areas/Admin/Views/` directories.

**Step 1 — Area ViewImports (Pitfall #5):**
- `Areas/Admin/Views/_ViewImports.cshtml`: Add `@using PetPlatform.Host.MVC`, `@using PetPlatform.Infrastructure.Identity`, `@addTagHelper *, Microsoft.AspNetCore.Mvc.TagHelpers`. Without this, Tag Helpers don't work in Admin views.

**Step 2 — Admin Layout:**
- `Areas/Admin/Views/Shared/_AdminLayout.cshtml`: Simple admin layout with Tailwind classes — sidebar navigation (Dashboard, Users, Roles), main content area, link back to home. Uses `dir="rtl"` consistent with root layout.

**Step 3 — DashboardController:**
- `Areas/Admin/Controllers/DashboardController.cs`:
  - `[Area("Admin")]`
  - `[Authorize(Policy = "Permission:Users.View")]` — requires Users.View permission (ADMN-01)
  - `Index()` GET: Returns admin dashboard view with basic stats placeholder (total users count, total roles count). Inject UserManager<ApplicationUser> and RoleManager<ApplicationRole> to get counts.

**Step 4 — UserController (ADMN-01, ADMN-02, ADMN-03):**
- `Areas/Admin/Controllers/UserController.cs`:
  - `[Area("Admin")]`
  - `[Authorize(Policy = "Permission:Users.View")]` on class (all actions require view permission)
  - Inject `UserManager<ApplicationUser>`, `RoleManager<ApplicationRole>`
  - `Index()` GET: Lists all users with their roles. Returns paginated list (use Skip/Take for basic pagination). Shows username, email, roles, lockout status.
  - `Details(string id)` GET: Shows user details — email, roles, lockout end date, creation date
  - `Activate(string id)` POST: Sets LockoutEnd to null (unlocks account). Requires `"Permission:Users.Manage"` policy. Per ADMN-02.
  - `Deactivate(string id)` POST: Sets LockoutEnd to DateTimeOffset.MaxValue (locks permanently). Requires `"Permission:Users.Manage"` policy. Per ADMN-02.
  - `AssignRole(string id)` GET: Shows form with available roles (from RoleManager) and current user roles
  - `AssignRole(string id, string role)` POST: Adds user to role via UserManager.AddToRoleAsync. Requires `"Permission:Users.Manage"` policy. Per ADMN-03.
  - `RemoveRole(string id, string role)` POST: Removes user from role via UserManager.RemoveFromRoleAsync. Requires `"Permission:Users.Manage"` policy.

**Step 5 — RoleController (ADMN-04, ADMN-05):**
- `Areas/Admin/Controllers/RoleController.cs`:
  - `[Area("Admin")]`
  - `[Authorize(Policy = "Permission:Roles.Create")]` on class
  - Inject `RoleManager<ApplicationRole>`
  - `Index()` GET: Lists all roles with their claim count. Per ADMN-04.
  - `Create()` GET: Returns create role form (name + description)
  - `Create(string name, string description)` POST: Creates role via RoleManager.CreateAsync(new ApplicationRole { Name = name, Description = description }). Per ADMN-04.
  - `Details(string id)` GET: Shows role details with assigned claims (permissions). Per ADMN-05.
  - `AddClaim(string roleId)` GET: Shows form to add a permission claim (text input for claim value)
  - `AddClaim(string roleId, string claimValue)` POST: Adds claim via RoleManager.AddClaimAsync(role, new Claim("Permission", claimValue)). Per ADMN-05.
  - `RemoveClaim(string roleId, string claimId)` POST: Removes claim via RoleManager.RemoveClaimAsync.

Verify by building: `dotnet build PetPlatform.sln`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Admin Controllers | Admin operations — must verify authorization before any mutation |
| Admin Area → Identity APIs | Direct UserManager/RoleManager calls — trust boundary for user/role data |
| Admin → User accounts | Activate/deactivate operations affect authentication state |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | Admin area access | critical | mitigate | Every Admin controller has [Authorize(Policy="Permission:...")] — non-admin users get 403 |
| T-03-02 | Tampering | Role permission assignment | high | mitigate | Only users with Roles.Create policy can create roles; only with Users.Manage can assign roles |
| T-03-03 | Information Disclosure | User list exposure | medium | mitigate | Only users with Users.View policy can access user list; email addresses shown but no passwords |
| T-03-04 | Elevation of Privilege | Account activation bypass | high | mitigate | Deactivate uses DateTimeOffset.MaxValue (permanent lockout); only Users.Manage policy holders can toggle |
| T-03-05 | Tampering | CSRF on admin forms | high | mitigate | All POST actions require [ValidateAntiForgeryToken], all forms include @Html.AntiForgeryToken() |
</threat_model>
