Create the Application layer services, contracts, and file storage abstraction. This task follows the Interface-First ordering pattern — define contracts first, then implement.

**Step 1 — Service interfaces:**
- `src/PetPlatform.Application/Interfaces/IPetService.cs`: Define methods: `Task<Result<PetDto>> CreateAsync(CreatePetDto dto, string ownerId)`, `Task<IEnumerable<PetDto>> GetAllByOwnerAsync(string ownerId)`, `Task<PetDto?> GetByIdAsync(int id)`, `Task<Result<PetDto>> UpdateAsync(int id, UpdatePetDto dto, string ownerId)`, `Task<Result<bool>> DeleteAsync(int id, string ownerId)`. Use `Result<T>` from Application/Common/Result.cs.
- `src/PetPlatform.Application/Interfaces/ICustomerService.cs`: Define methods: `Task<CustomerProfileDto> GetProfileAsync(string userId)`, `Task<Result<CustomerProfileDto>> UpdateProfileAsync(string userId, UpdateCustomerProfileDto dto)`
- `src/PetPlatform.Application/Interfaces/IFileStorageService.cs`: Define method: `Task<string> SaveFileAsync(IFormFile file, string subDirectory)` — accepts an IFormFile and subdirectory name, returns the relative path to the saved file. Per RESEARCH.md Architectural Responsibility Map: file upload belongs in Infrastructure tier.

**Step 2 — DTOs:**
- `src/PetPlatform.Application/DTOs/PetDto.cs`: Record or class with Id, Name, Species (PetSpecies enum), Breed, Age, Weight, PhotoPath, OwnerId, CreatedAt. Include CreatePetDto (Name, Species, Breed, Age, Weight, Photo as IFormFile) and UpdatePetDto (Name, Breed, Age, Weight).
- `src/PetPlatform.Application/DTOs/CustomerProfileDto.cs`: Record with Id, UserId, FullName, Address, Phone, City, NotificationPreferences. Include UpdateCustomerProfileDto (FullName, Address, Phone, City, NotificationPreferences).

**Step 3 — FluentValidation validators:**
- `src/PetPlatform.Application/Validators/CreatePetValidator.cs`: Per RESEARCH.md code example — NotEmpty + MaxLength(100) for Name, IsInEnum for Species, InclusiveBetween(0,50) for Age, GreaterThanOrEqualTo(0) + LessThanOrEqualTo(500) for Weight. Use `AbstractValidator<CreatePetDto>`.
- `src/PetPlatform.Application/Validators/UpdateCustomerProfileValidator.cs`: NotEmpty + MaxLength(100) for FullName, MaxLength(20) for Phone, MaxLength(100) for City. Use `AbstractValidator<UpdateCustomerProfileDto>`.

**Step 4 — Service implementations:**
- `src/PetPlatform.Application/Services/PetService.cs`: Implement IPetService. Inject `ApplicationDbContext` (or IDbContextFactory<ApplicationDbContext>). Use FluentValidation validators to validate before persist. Ownership enforcement: when querying/mutating, always filter by ownerId parameter. Return `Result<T>.Success` on valid operations, `Result<T>.Failure` on validation errors. Use Guard.AgNullOrWhiteSpace for input validation on service parameters.
- `src/PetPlatform.Application/Services/CustomerService.cs`: Implement ICustomerService. Inject `ApplicationDbContext`. GetOrCreate pattern: if profile doesn't exist for userId, create empty one on first access. Update only provided fields.

**Step 5 — File storage service (per RESEARCH.md Architectural Responsibility Map):**
- `src/PetPlatform.Infrastructure/Services/FileStorageService.cs`: Implement `IFileStorageService`. Inject `IWebHostEnvironment`. In `SaveFileAsync`: validate extension whitelist (.jpg, .jpeg, .png, .webp), validate size (5MB max), generate server-side filename (`Guid.NewGuid() + extension`), create subdirectory if needed under `WebRootPath`, write file to disk using `FileStream`, return relative path (`/uploads/{subDirectory}/{filename}`). Per RESEARCH.md Pattern 5 — but placed in Infrastructure tier, not controller. Throw `InvalidOperationException` with descriptive message on validation failure (extension mismatch or size exceeded).

Register services in DI: Add `services.AddScoped<IPetService, PetService>()`, `services.AddScoped<ICustomerService, CustomerService>()`, and `services.AddScoped<IFileStorageService, FileStorageService>()` in Program.cs (edit existing file).

Verify by building: `dotnet build PetPlatform.sln`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → MyPetsController | Pet creation/edit forms — untrusted input (name, species, file upload) |
| Browser → MyAccountController | Profile edit form — untrusted input (address, phone) |
| MyPetsController → File System | Photo upload crosses into server storage — path traversal risk |
| Controller → PetService | Ownership enforcement — must verify user owns pet before mutation |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Tampering | Pet photo upload (FileStorageService) | high | mitigate | Extension whitelist (.jpg/.jpeg/.png/.webp), 5MB size limit, server-generated filenames (Guid.NewGuid()) — validation and storage in Infrastructure tier via IFileStorageService, per RESEARCH.md Pattern 5 and Architectural Responsibility Map |
| T-02-02 | Elevation of Privilege | Pet ownership bypass | high | mitigate | PetService enforces OwnerId match on all mutations (Update/Delete), controller passes current user ID |
| T-02-03 | Information Disclosure | Public pet view (PetController) | low | accept | PET-03 requires public view — intentional by design, no sensitive data exposed |
| T-02-04 | Tampering | CSRF on profile/pet forms | high | mitigate | @Html.AntiForgeryToken() on all forms, [ValidateAntiForgeryToken] on all POST actions |
| T-02-05 | Injection | Pet name/breed fields | medium | mitigate | EF Core parameterized queries prevent SQL injection; Razor auto-encoding prevents XSS |
</threat_model>
